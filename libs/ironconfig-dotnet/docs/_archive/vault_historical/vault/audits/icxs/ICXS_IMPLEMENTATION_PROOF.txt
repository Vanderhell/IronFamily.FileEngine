IMPLEMENTATION DETAILS
================================================================================

ICXS File Format:
-----------------
Byte Layout:
  [0-63]:       Header (64 bytes fixed)
                - Magic: "ICXS" (4 bytes)
                - Version: 0 (1 byte)
                - Flags: 0x00|0x01 for CRC (1 byte)
                - SchemaHash: SHA-256 first 16 bytes
                - SchemaBlockOffset: uint32 LE
                - DataBlockOffset: uint32 LE
                - CrcOffset: uint32 LE

  [64+]:        Schema Block
                - FieldCount: varint
                - Fields (sorted by ID):
                  - FieldId: uint32 LE
                  - FieldType: byte (1-5)

  [data+]:      Data Block
                - RecordCount: uint32 LE
                - RecordStride: uint32 LE
                - Fixed Region: recordCount * recordStride bytes
                  - Each record: field values in ID order
                  - i64/u64/f64: 8 bytes LE
                  - bool: 1 byte (0x00/0x01)
                  - str: 4 bytes LE offset to variable region
                - Variable Region: concatenated strings
                  - Each string: 4-byte length LE + UTF-8 bytes

  [crc?]:       CRC32 (optional, 4 bytes)
                - Computed over [0 .. crcOffset)
                - IEEE polynomial 0x04C11DB7

Key Properties:
- Zero-copy: No dictionary allocation at read time
- O(1) field access: Calculate offset from field ID + fixed sizes
- Deterministic: Fixed byte output for same input
- Validated: Schema hash + CRC protection

ARCHITECTURE & DESIGN DECISIONS
================================================================================

1. Zero-Copy Design:
   Problem: Traditional approaches allocate dictionaries at read time
   Solution: Calculate field offsets on-the-fly using field ID + fixed sizes
   Benefit: O(1) access without allocation overhead

2. Three-Pass Encoding:
   Problem: CRC must be computed after header is finalized
   Solution:
     - Pass 1: Collect all variable data with offsets
     - Pass 2: Write fixed region using pre-calculated offsets
     - Pass 3: Write variable region
     - Then: Finalize header, compute CRC
   Benefit: Deterministic output, correct CRC computation

3. Schema Separation:
   Problem: How to avoid storing key names in every record?
   Solution: External schema file maps field IDs to names and types
   Benefit: Reduces file size, enables type checking at read time

4. Deterministic Schema Hash:
   Problem: Schema needs verification without external dependencies
   Solution: Canonical JSON (no whitespace, sorted keys) â†’ SHA-256 first 16 bytes
   Benefit: Unique, stable hash independent of serialization library

5. Field Stride Calculation:
   Problem: How to locate fields without searching?
   Solution: Pre-calculate stride, store in header
   Cost: 4 bytes in header
   Benefit: O(1) record access
