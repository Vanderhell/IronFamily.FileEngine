===============================================================================
Command:
  dotnet run -- validate min.icfg

Result:
  ✓ BJV2 valid
  ✓ File is valid
  Exit code: 0

Output: min.icfg passes validation (as expected)

===============================================================================
TEST 2: CORRUPT ONE BYTE (XOR 0xFF at position 45/90)
===============================================================================
Original byte at position 45: 0x00
Corrupted byte at position 45: 0xFF
Output file: min_corrupt.icfg

This byte is in the middle of the file content (after header+dict+vsp headers,
inside the root value encoding).

===============================================================================
TEST 3: VALIDATE CORRUPTED FILE
===============================================================================
Command:
  dotnet run -- validate min_corrupt.icfg

Result:
  ✓ BJV2 valid
  ✓ File is valid
  Exit code: 0

Output: min_corrupt.icfg also passes validation despite corruption!

===============================================================================
CONCLUSION
===============================================================================
FINDING: CRC is NOT validated on read.

EVIDENCE:
1. File header claims CRC is present ("CRC: present")
2. One byte of file data was corrupted (position 45: 0x00 → 0xFF)
3. Validation should have FAILED if CRC was being checked
4. Validation PASSED without detecting corruption

This demonstrates conclusively that:
- The CRC flag is SET in the header when --crc on is used
- But NO actual CRC computation or validation is performed
- Corrupted files are accepted as valid
