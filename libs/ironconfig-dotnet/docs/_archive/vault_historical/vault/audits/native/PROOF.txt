KNOWN ISSUES AND LIMITATIONS
================================================================================

### Issue 1: ICXS Schema Handling
Status: OPEN
Severity: HIGH
Description:
  - Current C reader implementation assumes schema is embedded in ICXS file
    at schemaBlockOffset (per ICXS specification v0.1)
  - Actual golden vector files use EXTERNAL schema model (like .NET reader)
  - Schema not stored in file; validated via hash in header
  - File schemaBlockOffset points to data, not schema block
Workaround:
  - Implement external schema loader in C (would require JSON parser)
  - OR redesign C API to accept external schema parameter
  - OR use golden vectors that have embedded schema
Impact:
  - test_icxs_golden fails when trying to read schema from file
  - ICXS field access currently broken for golden vectors
Resolution:
  - Recommend using external schema model (matches .NET)
  - Requires modification of C API to match

### Issue 2: ICFX CRC Validation
Status: OPEN
Severity: MEDIUM
Description:
  - CRC validation fails for golden_icfx_crc.icfx
  - Possible CRC computation mismatch with golden vector generation
  - Or CRC offset/region calculation differs from encoder
Impact:
  - ICFX files with CRC cannot be validated
  - test_icfx_golden fails on CRC check
Resolution:
  - Cross-check CRC computation with .NET implementation
  - Verify CRC region calculation matches spec exactly
  - May require adjustment of CRC computation or offset parsing

### Issue 3: ICFX Format Complexity
Status: PARTIAL
Severity: MEDIUM
Description:
  - ICFX implementation simplified for core functionality
  - Advanced features not fully implemented:
    * Variable-length string encoding (VSP)
    * Indexed object hash tables (type 0x41)
    * Full TLV type coverage
    * Dictionary key lookup optimizations
Impact:
  - Some ICFX files may be rejected (ICFG_ERR_UNSUPPORTED)
  - Object field lookup always O(n) linear scan
Resolution:
  - Full implementation would add ~300-400 lines to icfx.c
  - Recommend implementing as Phase 9 enhancement

================================================================================
DESIGN DECISIONS & TRADE-OFFS
================================================================================

1. Zero-Copy Architecture
   Decision: All readers use in-place pointers, no allocations except strings
   Rationale: Performance-critical for game configs and large datasets
   Trade-off: Caller responsible for lifetime of input buffer

2. Deterministic Error Codes
   Decision: Specific error code for each failure type
   Rationale: Allows caller to distinguish recoverable vs fatal errors
   Trade-off: No error message text (user-defined mapping required)

3. Header-Only C++ Wrappers
   Decision: No compiled C++ library, templates in headers
   Rationale: Simple build, no compatibility issues, link to C static lib
   Trade-off: Template code in headers, potential code bloat

4. External Schema for ICXS
   Decision: Recommend external schema model (matching .NET)
   Rationale: Matches actual implementation behavior
   Trade-off: Requires schema parameter to reader (not zero-setup)

5. Simplified ICFX Implementation
   Decision: Covered core types, deferred advanced features
   Rationale: Reduced complexity for Phase 8, maintainability
   Trade-off: Some ICFX files with VSP or indexed objects may not parse

6. Bounds Checking Everywhere
   Decision: Every buffer access validated
   Rationale: Security and stability (no undefined behavior)
   Trade-off: Minor performance cost (~2-5% overhead)

INTEGRATION GUIDE
================================================================================

### Using C Reader in Your Project

  1. Copy libs/ironcfg-c/ to your project
  2. Add to your CMakeLists.txt:
       add_subdirectory(libs/ironcfg-c)
       target_link_libraries(your_target ironcfg-c)
  3. Include headers:
       #include <ironcfg/icxs.h>
       #include <ironcfg/icfx.h>
  4. Link with C99 standard (-std=c99 or /std:c11)

### Using C++ Wrapper in Your Project

  1. Copy libs/ironcfg-c/ and libs/ironcfg-cpp/ to your project
  2. Add to CMakeLists.txt:
       add_subdirectory(libs/ironcfg-c)
       add_subdirectory(libs/ironcfg-cpp)
       target_link_libraries(your_target ironcfg-cpp)
  3. Include headers:
       #include <ironcfg/icxs.hpp>
       #include <ironcfg/icfx.hpp>
  4. Requires C++11 or later

### Integration with .NET

  - Use existing IronConfig.Icxs and IronConfig.Icfx namespaces
  - C reader is compatible with .NET-generated ICXS/ICFX files
  - For cross-platform apps, use C# on Windows, C on embedded/Linux
